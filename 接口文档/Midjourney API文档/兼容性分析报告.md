## 二、请求参数传递机制

new-api 对 Midjourney 请求采用 **透传转发** 策略：

```
Client Request Body
       │
       ▼
DoMidjourneyHttpRequest (service/midjourney.go:165)
  ├── 反序列化为 map[string]interface{}
  ├── 按开关处理 accountFilter / notifyHook 删除
  ├── 按开关处理 prompt 中 --fast/--relax/--turbo 清理
  ├── 其余字段完整保留
  └── 重新序列化后转发给上游
```

`MidjourneyRequest` DTO 仅用于业务逻辑提取（`Prompt`、`TaskId`、`Action`、`Index` 等），文档中 `mode`、`dimensions`、`botType`、`motion`、`image` 等字段即使不在 DTO 中也会透明传递到上游。

**结论：所有文档描述的参数都会被原样转发，不会丢失。**

## 四、已识别的兼容性问题

### 🔴 问题一：上传接口 `result` 字段类型不匹配

**涉及文件**：`service/midjourney.go:231-249`、`relay/mjproxy_handler.go:526-630`

**问题描述**：
- 文档定义 `POST /mj/submit/upload-discord-images` 返回 `result` 为 **`[]string`**（图片 URL 数组）
- `MidjourneyResponse.Result` 为 **`string`** 类型
- `DoMidjourneyHttpRequest` 虽有 `MidjourneyUploadResponse`（`Result []string`）作 fallback，但返回值固定使用 `midjResponse`，upload 的 fallback 结果未被使用

**影响链路**：
```
上游返回: {"code":1, "result":["url1","url2"]}
       │
       ▼
json.Unmarshal → MidjourneyResponse 失败（array → string 类型不匹配）
       │
       ▼
fallback → MidjourneyUploadResponse 成功
       │
       ▼
但 midjResponse 仍是零值 {Code: 0, Result: ""}
       │
       ▼
RelayMidjourneySubmit: Code=0 → 判定为错误 → 不计费
DB 记录 MjId=""、FailReason="success"（实际是成功）
```

**用户侧影响**：
- 客户端收到的原始响应正确（透传），图片能正常上传
- DB 中上传任务的 `MjId` 为空
- 任务被误标记为失败

---

### 🟡 问题二：任务查询 DTO 缺少 `imageHeight`、`imageWidth`、`imageUrls`

**涉及文件**：`dto/midjourney.go:47-68`、`model/midjourney.go`

**问题描述**：

文档 `指定id查询任务` 和 `指定id列表查询任务` 的响应中包含以下字段，但 new-api 的 `MidjourneyDto` 和 `Midjourney` DB 模型均未定义：

| 缺失字段 | 文档位置 | 说明 |
|---------|---------|------|
| `imageHeight` (int) | 文档 67-69 行 | 生成图片的高度 |
| `imageWidth` (int) | 文档 70-72 行 | 生成图片的宽度 |
| `imageUrls` ([]string) | 文档 98-102 行 | 多图场景下的图片 URL 列表 |

**影响**：
- `GET /mj/task/{id}/fetch` 和 `POST /mj/task/list-by-condition` 返回的 JSON 不包含这三个字段
- 由于 DB 未存储这些字段，即使上游回调中包含这些信息也无法持久化

---

### 🔵 问题三：SwapFace 请求格式（文档 schema 与 example 不一致）

**涉及文件**：`dto/midjourney.go:10-13`

**文档状态**：
- Schema 定义为 `multipart/form-data`（`source`/`target` 为 binary）
- Example 实际使用 `sourceBase64`/`targetBase64`（JSON body）

**new-api 实现**：`SwapFaceRequest` 使用 JSON body 接收 `sourceBase64` / `targetBase64`。

**结论**：代码与文档 example 一致，属于文档 schema 描述不严谨，无实际影响 🟢。
